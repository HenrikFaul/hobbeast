# v1.5.2 AI Dev Prompts — 15041003

## Scope
Stabilize the currently buggy admin/user-generation flows without regressing existing working functionality.

## Required implementation goals
1. Fix `mass-create-users` so auth user creation and profile enrichment are separated.
2. Remove reliance on fragile `upsert(... onConflict ...)` for profile persistence.
3. Normalize hobbies to `string[]` and persist them to `profiles.hobbies`.
4. Fix admin bulk selection so filtered selection count and single-row deselect work correctly.
5. Use `profile.id` as the canonical selection key in the UI.
6. Add/service-deploy `admin-bulk-user-actions` for preview/apply flows.
7. Soften `has_role`, notifications, and notification-preferences failures.
8. Prevent startup generic edge-function toasts from hidden admin panels.
9. Replace risky event-count HEAD logic with safe select-length logic.
10. Keep changes minimal in scope and regression-aware.

## Regression guardrails
- Do not break existing auth flows.
- Do not remove working admin tabs.
- Do not redesign unrelated screens.
- Do not silently delete prior changelog or lessons content.
- Keep new DB changes append-only and idempotent.

## Validation checklist
- Bulk user generation creates auth users and enriches profiles.
- `profiles.hobbies` is filled after generation.
- Error toast no longer falsely says `0` when users were created.
- Filter-based selection count matches visible selected profiles.
- Deselecting one row does not clear all rows.
- `admin-bulk-user-actions` deploys successfully.
- Organizer mode no longer throws `events ... 500` from owned count query.
- Notification preferences and has_role failures do not white-screen or break admin page.
