# Business Request Summary — v1.5.2 (15041003)

## Business goal
Stabilize the newly introduced admin batch user operations and simulated user generation so the admin can reliably generate, classify, filter, select, activate/deactivate, and delete users without misleading UI feedback or backend/runtime instability.

## Main pain points observed
- Generated users were partially created but UI feedback remained misleading.
- Hobby values were not reliably saved with generated users.
- Batch profile selection state was inconsistent.
- Deselecting a single row could clear the full selection.
- Admin-side event ownership count queries produced runtime failures.
- Hidden background admin integrations could show noisy generic error toasts.

## Expected result
- Generated users should be created consistently.
- Their profiles must be enriched with hobbies and marked as generated users.
- Admin batch selection must use stable row-level identifiers.
- Bulk actions must operate on the intended selected profiles only.
- Organizer/admin event count logic must avoid the current failing request pattern.
- Non-critical admin background failures should degrade gracefully.

## Protected existing behavior
- Existing authentication and admin access rules must remain intact.
- Existing admin tabs must remain available.
- Existing user profile and event structures must not be destructively rewritten.
