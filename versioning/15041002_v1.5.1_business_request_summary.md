# Hobbeast business request summary

**Versioning ID:** 15041002  
**Version:** v1.5.1

## Scope
This package addresses the currently observed admin/event regressions and adds admin management capabilities for generated users.

## Required fixes
1. Stabilize mass user generation so generated users are actually created, are marked as generated users, and their hobbies persist to `profiles`.
2. Stop false toast messages that report zero created users when auth users already exist.
3. Remove event list/detail/admin recursion issues causing 500 errors around `event_participants`.
4. Harden admin-role and notification related fetch flows so missing rows do not surface as noisy UI failures.

## New admin requirements
1. Profiles must show whether the user is a real user or a generated/artificial user.
2. Admin must be able to batch select profiles via a filter-based bulk-selection flow.
3. Filters must support:
   - real vs generated
   - registration age in days
   - last login age in days
   - whether the user owns an open event
4. Bulk actions required now:
   - delete
   - activate
   - deactivate
   - reminder action placeholder visible but inactive
5. Batch delete must require confirmation.

## Data model additions
1. `profiles` needs generated-vs-real marker and activation state.
2. Event-side admin metrics fields are required for:
   - creator
   - cancelled vs held outcome
   - registrations
   - cancellations
   - attended count
   - rating summary

## Non-negotiable constraints
- No regression in already working flows.
- Keep admin/auth/event/profile behavior coherent.
- Append changelog and lessons only; do not overwrite history.
- Deliver patch-only style artifacts plus versioning files.
